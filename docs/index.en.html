Guide

Welcome to the GOPAX REST API documentation. You can utilize the REST API to place orders and get a detailed price and market feeds.

REST API comprises of private API where you authenticate via auth-token and check orders and balances, and public API where recent trades and prices.

API Endpoint (URL)

GOPAX REST API provides endpoints for account/order management and public market data.

https://api.gopax.com

API Call Rate Limit

When the API call rate limit is exceeded, the status code 429 - Too Many Requests will be returned.

The rate is limited per IP for Public API and per API Key for Private API. At most 20 calls can be made within the 1 second moving window, respectively. For a specific type of order, you can call them twice during the aforementioned 1 second window.

Request & Response Format

The content-type for all requests and responses is application/json, and follows typical HTTP response status codes. For example, a successful request will return status code 200.

Private API Authentication

In order to authenticate for the Private API, the following HTTP header must be included in the REST request.

  1. API-KEY: the issued API Key
  2. SIGNATURE: the message signature value (further description below)
  3. NONCE: a constantly increasing non-redundant value (typically, timestamp)
You can issue two or more API keys.
If identical NONCE values are used, the server will reject the request.

SIGNATURE formation procedure

Example :

const crypto = require('crypto');

const generateSignature = (secret, nonce, method, path, body = null) => {
  const requestPath = path.split('?')[0];
  const _body = body ? JSON.stringify(body) : '';

  const what = `${nonce}${method}${requestPath}${_body}`;
  const key = Buffer(secret, 'base64');
  const hmac = crypto.createHmac('sha512', key);
  return hmac.update(what).digest('base64');
};

function generateSignature($secret, $nonce, $method, $path, $body = null)
{
  $_body = $body ? json_encode($body) : '';
  $tokenizedPath = explode('?', $path);
  $requestPath   = $tokenizedPath[0];
  $what          = $nonce . $method . $requestPath . $_body;
  $secret        = base64_decode($secret);
  return base64_encode(hash_hmac('sha512', $what, $secret, true));
}

  1. Perform string-concatenation of the following in order.
    1. NONCE of the header
    2. HTTP method (ALL CAPS): 'GET', 'POST', 'DELETE' or such
    3. API endpoint URL (e.g. '/orders', '/trading-pairs/ETH-IDR/book')
    4. Request parameter body in JSON format (if not applicable, do not add any string)
  2. Decode the issued secret in base64.
  3. Use the value from 2. as the secret key and sign as sha512 HMAC.
  4. Encode the value from 3. in base64.
The Secret Key is generated when the API key is issued. If the Secret Key is lost, you must reissue the API Key.

private apiKey = '';
private apiSecret = '';

const API_HOST = 'API Endpoint URL';
const VERSION = 'gopax-php-sdk-20171216';

private function request(string $method, string $path, $request = NULL)
{
    $curl = curl_init();

    $mt = explode(' ', microtime());
    $nonce     = $mt[1] . substr($mt[0], 2, 6);
    $method    = strtoupper($method);

    $tokenizedPath = explode('?', $path);
    $requestPath   = $tokenizedPath[0];
    $what          = $nonce . $method . $requestPath . $request;
    $secret        = base64_decode($this->apiSecret);
    $signature     = base64_encode(hash_hmac('sha512', $what, $secret, true));

    $headers[] = 'Content-Type: application/json';
    $headers[] = 'API-KEY: ' . $this->apiKey;
    $headers[] = 'SIGNATURE: ' . $signature;
    $headers[] = 'NONCE: ' . $nonce;
    curl_setopt($curl, CURLOPT_USERAGENT, self::VERSION);
    curl_setopt($curl, CURLOPT_URL, self::API_HOST . $path);
    curl_setopt($curl, CURLOPT_RETURNTRANSFER, true);
    curl_setopt($curl, CURLOPT_HTTPHEADER, $headers);
    curl_setopt($curl, CURLOPT_POST, TRUE);
    curl_setopt($curl, CURLOPT_POSTFIELDS, $postData);

    $json       = curl_exec($curl);

}

Forced Completion Reason Reason for completing an order with remainingAmount larger than zero
You can now place a stop order by denoting a stopPrice value greater than zero. Stop order is an order functionality where if current price matches or exceeds (in either directions) the triggering price (stopPrice), an order of your designation (GTC-type limit order or market order) is executed. Stop order is possible in both buy and sell order with both price movements (up or down) supported. Stop orders are mainly used for cutting losses (stop-loss), realizing profit (take-profit), and buying when price seems appropriate (stop-entry). Stop order changes its type to a regular order after being triggered, and if you have placed a stop order using clientOrderId, then you can use the same ID to query and/or cancel the order.
Client Order ID is a custom order id generated by user, separate from the order id. User can utilize clientOrderId when posting, getting, and deleting an order. If active order is fully matched or cancelled, user will be able to use associated clientOrderId for an hour in querying the completed status of the order. You can use the same clientOrderId for a different order after an hour from order completion.
PO(Post Only) only places an entire order, and cancels the order when there are corresponding orders present for matching.
IOC(Immediate-Or-Cancel Order) matches a given order for an amount coverable upto the given price. After matching, if there's still an amount remaining, rest of the order will be immediately cancelled.
FOK(Fill-Or-Kill Order) matches a given order if and only if all of the order amount can be matched for the given price. Otherwise, all order amount is immediately cancelled.
Protection, if set to 'yes', will protect a given order by cancelling remaining amount if the price is over 10% of first matching price. Protection can be used for both limit and market order.
forcedCompletionReason is denoted when using order protection and/or a specific limit order type (PO, IOC, FOK) and order has completed with remainingAmount larger than zero. Two possible values are timeInForce and protection.
If an order is a market order and the request contained limit order type, such type specification is ignored and handled as a market order. FOK, IOC limit order types cannot be used with Protection.
amount in market order is in asset that the user is paying (e.g. ETH-KRW bid order will use KRW as an amount, ETH for an ask order).
